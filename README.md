# five-bot-analysis-pipeline (FBA)

A fully local, multi-agent data analysis pipeline for condition monitoring of rotating machinery — and a template for any domain where structured data → expert report is a repeatable workflow.

**Five specialist bots. No cloud. No API keys. All inference runs on local hardware.**

---

## Architecture

```
User request
     │
     ▼
┌─────────┐     ┌───────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│ Planner │ ──► │ Coder │ ──► │ Analyzer │ ──► │ Reporter │ ──► │ Reviewer │
└─────────┘     └───────┘     └──────────┘     └──────────┘     └──────────┘
  Reads data      Runs         Interprets        Writes            QA checks
  files, plans   scripts,      Coder output,     client            report
  the work       reports       finds patterns    report            against
                 output                                            source data
```

Each bot is a separate Hermes Agent profile with its own local LLM. Bots communicate via handover prompts — structured text passed from one session to the next. There is no shared memory or live inter-bot communication: every handover is explicit and auditable.

### Bot roles

| Bot | Responsibility | Does NOT do |
|-----|----------------|-------------|
| **Planner** | Reads asset background files, scopes the analysis, produces a numbered task plan | Write code or perform analysis |
| **Coder** | Runs Python analysis scripts, reports exact output | Interpret results or modify scripts |
| **Analyzer** | Interprets Coder output — finds patterns, trends, anomalies | Compute or transform raw data |
| **Reporter** | Converts Analyzer findings into a client-facing report | Add findings not in the Analyzer output |
| **Reviewer** | Verifies the report against source data and Analyzer findings | Perform analysis or rewrite content |

## Demo dataset — Vesi Oy Pump Station 3

The included demo simulates bearing fault progression in a Grundfos CM5-6 centrifugal pump at a municipal water treatment plant in Tampere, Finland.

**All data is fully synthetic.** It was generated by `generate_data.py` to demonstrate the complete pipeline workflow. No real measurements, no real client.

The dataset covers four monitoring sessions (Sep–Dec 2024) progressing through four fault stages:

| Session | Date | Fault stage | DE bearing temp | BPFO depth |
|---------|------|-------------|-----------------|------------|
| 1 | 2024-09-15 | Healthy | ~52.9 °C | none |
| 2 | 2024-10-17 | Incipient | ~55.4 °C | 0.018 |
| 3 | 2024-11-14 | Developing | ~57.5 °C | 0.120 |
| 4 | 2024-12-12 | Severe | ~59.6 °C | 0.450 |

**Data files produced by `generate_data.py`:**

```
background/          (committed — pre-generated for reference)
  pump_info.json         Asset spec, bearing parameters, fault frequencies
  measurement_setup.json Sensor config, DAQ settings, calibration metadata
  site_info.json         Site details, maintenance history, baseline values

data/               (git-ignored — run generate_data.py to create)
  acoustic/
    acoustic_2024-09-15.wav    1-channel PCM_16, 48 kHz, 60 s
    acoustic_2024-10-17.wav
    acoustic_2024-11-14.wav
    acoustic_2024-12-12.wav
    measurement_log.json       Session metadata + known fault frequencies
  automation/
    scada_data.csv             10-minute time series, Sep–Dec 2024
```

**Known discrepancy:** `measurement_setup.json` specifies a 24-bit DAQ (`resolution_bits: 24`), but the generated WAV files are 16-bit PCM (`scipy.io.wavfile` writes int16). Task 1 flags this. It reflects a real-world scenario where metadata and actual recordings can diverge.

## Prerequisites

- [Hermes Agent](https://github.com/roy-ht/hermes) — multi-profile local agent framework
- [Ollama](https://ollama.com) — local LLM runtime
- Python 3.10+ with a virtual environment

**Recommended models (tested on 8 GB VRAM):**

| Bot | Model | Notes |
|-----|-------|-------|
| Planner | qwen3:14b | Good at reading structured files and planning |
| Coder | qwen3:8b | Instruction-following; runs scripts reliably |
| Analyzer | qwen3:14b | Needs reasoning depth for spectral interpretation |
| Reporter | qwen3:14b | Structured writing; `reasoning_effort` not needed |
| Reviewer | qwen3:14b | Fact-checking against source data |

qwen3 variants are strongly preferred — they follow structured output formats reliably at these sizes. deepseek-r1 is an alternative for Reviewer but is more prone to hallucination at 8b.

## Setup

### 1. Clone and install Python dependencies

```bash
git clone https://github.com/<your-username>/five-bot-analysis-pipeline.git
cd five-bot-analysis-pipeline
python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt
```

### 2. Generate the synthetic dataset

```bash
python generate_data.py
```

This creates `background/` (if not already committed) and `data/`. Takes about 30 seconds.

### 3. Install Hermes profiles

Copy the bot profile directories to your Hermes profiles folder. By default this is `~/.hermes/profiles/`:

```bash
for bot in planner coder analyzer reporter reviewer; do
    mkdir -p ~/.hermes/profiles/$bot
    cp bot_profiles/$bot/SOUL.md ~/.hermes/profiles/$bot/SOUL.md
done
```

### 4. Configure Modelfiles

Each bot needs an Ollama Modelfile. Example for the Analyzer (adjust `FROM` to your model):

```
FROM qwen3:14b
PARAMETER num_ctx 65536
PARAMETER temperature 0.3
SYSTEM """/no_think
"""
```

**Critical:** for any qwen3 or deepseek-r1 model, add `/no_think` as the **first characters** of the `SYSTEM` block and **do not** include `PARAMETER stop "/think"`. That stop token terminates generation before any visible output is written.

Build each model:

```bash
ollama create analyzer-model -f Modelfile-analyzer
ollama create reporter-model -f Modelfile-reporter
# ... etc.
```

### 5. Set `terminal.cwd` in each bot's `config.yaml`

Hermes auto-loads `AGENTS.md` from `terminal.cwd`. Point every bot at this project:

```yaml
terminal:
  cwd: /path/to/five-bot-analysis-pipeline
```

Copy `AGENTS.md` to the project root on your machine and update the paths inside it.

## Running the pipeline

### Standard run (all four sessions)

**Step 1 — Planner:** Send the handover prompt from `bot_profiles/planner/handover_prompt.md`

**Step 2 — Coder:** Run the four scripts in order:
```
run scripts/task1_wav_inspect.py
run scripts/task2_psd.py
run scripts/task3_fault_amplitudes.py
run scripts/task4_scada_context.py
```

**Step 3 — Analyzer:** Send the handover prompt from `bot_profiles/analyzer/handover_prompt.md`

**Step 4 — Reporter:** Send the handover prompt from `bot_profiles/reporter/handover_prompt.md`

**Step 5 — Reviewer:** Send the handover prompt from `bot_profiles/reviewer/handover_prompt.md`

### Analysis scripts

| Script | Inputs | Output | What it does |
|--------|--------|--------|--------------|
| `task1_wav_inspect.py` | WAV files, `measurement_setup.json` | `wav_inspection.json` | Validates WAV headers; flags sample rate, channel count, and bit-depth discrepancies |
| `task2_psd.py` | WAV files, `wav_inspection.json` | `psd_results.npz` | Welch PSD on channel 0 (ACC-DE) for all sessions |
| `task3_fault_amplitudes.py` | `psd_results.npz`, `pump_info.json` | `fault_amplitudes.csv` | Amplitude at bearing fault frequencies and harmonics |
| `task4_scada_context.py` | `scada_data.csv` | `scada_context.json` | SCADA ±6 h window around each measurement session |

## Adapting to a new asset

The SOUL files are generic — they define each bot's role and approach, not the asset. All asset-specific content lives in the handover prompts and the Planner's context section.

To adapt for a new asset or client:

1. Replace the background files (`background/`) with your asset's specs
2. Update `bot_profiles/planner/handover_prompt.md` with the new asset details, fault frequencies, and data file locations
3. Write or adapt analysis scripts for the new data format
4. Update `AGENTS.md` with the new project context

The SOUL files, Modelfiles, and Hermes configuration do not need to change.

## Key lessons learned

A few non-obvious things discovered while building this:

- **`PARAMETER stop "/think"` kills output** for qwen3/deepseek-r1. The model exhausts its token budget on internal reasoning, then hits the stop token before writing any visible response. Fix: remove it, use `/no_think` in the Modelfile SYSTEM block.
- **`terminal.cwd` controls which `AGENTS.md` loads.** Wrong `cwd` = wrong project context bleeding into every bot. Each project needs its own `AGENTS.md` and each bot's config must point to it.
- **Use `terminal`, not `execute_code`, for the Coder bot.** The `execute_code` sandbox is isolated from the project's virtual environment and lacks the required scientific Python packages.
- **`ollama_num_ctx` is the correct key** for context window size in Hermes config (not `ollama_num_context_length`).

## Roadmap

### Hermes Kanban orchestration
Hermes includes a Kanban feature that can track task state across bot sessions. The next step is wiring the pipeline into it so that each bot's completion automatically queues the next — eliminating the manual handover step and giving a visual overview of where a job is in the pipeline.

### Telegram group handovers
The current handover model is manual: the user copies the handover prompt and opens the next bot session. A more autonomous version would route handovers through a Telegram group shared by all five bots. Each bot posts its output to the group on completion; the next bot picks it up as its input. This turns the group chat into a lightweight orchestration bus and makes the pipeline observable in real time without a separate dashboard.
